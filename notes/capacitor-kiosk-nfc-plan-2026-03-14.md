# Capacitor Kiosk App — Justification & Plan

_2026-03-14_

## Why We're Here

Over two days of testing on the Samsung Galaxy Tab Active3 (Android 13, One UI 5.1.1, Chrome 145+), we exhausted every web-based approach to foreground NFC reading:

| Attempt | Result |
|---------|--------|
| `NDEFReader.scan()` in Chrome tab (auto-start) | `scan()` succeeds, `reading` event never fires |
| `NDEFReader.scan()` with user gesture trigger | Same — event never fires |
| NDEF text record instead of URL record | Android intercepts the tap before Chrome can read |
| Both URL + text records on the tag | `reading` still never fires in foreground |
| Revoke Chrome NFC permission to prevent interception | Foreground reads still blocked |
| Samsung Internet browser | Same behavior — foreground NFC blocked |
| Standalone PWA with `NDEFReader.scan()` | `scan()` succeeds, green "NFC active" indicator shows, `reading` event still never fires |

**Root cause**: Samsung One UI intercepts NFC events at the system level before delivering them to any WebView-based process — Chrome tabs, Samsung Internet, and standalone PWAs (which use Chrome's WebView under the hood). This is firmware behavior, not a bug in our code. The Web NFC API cannot work for foreground reads on this device family.

**What still works**: Android's NDEF URL dispatch (cold start) — when Chrome/the browser is fully closed, tapping a badge opens the URL and records the scan. But this requires closing the browser between every worker tap, which is not viable for a kiosk line.

## Why Capacitor

Android's native `enableForegroundDispatch()` API operates at the app process level, below Samsung's browser interception layer. A native app — even one whose UI is a WebView — registers for NFC through the Android Activity lifecycle, not through Chrome's Web NFC shim. This is the standard approach for NFC-heavy Android applications and works on all Samsung devices.

Capacitor is the right wrapper because:

1. **Reuses the existing React UI** — KioskPage's component structure, styling, API calls, and QR scanning all carry over. Only the NFC layer changes.
2. **Minimal new code** — the native NFC plugin handles `enableForegroundDispatch()` and bridges tag reads to JavaScript. We don't write Kotlin/Java ourselves.
3. **Future-proof** — we already plan a GPS oracle that will need native geofencing. A Capacitor shell positions us for that without starting over.
4. **No Play Store required** — sideload an APK for internal kiosk use.

## Same Repo or Separate Repo?

**Same repo, as a `kiosk-app/` directory.**

Reasons:

- The kiosk app is a thin native wrapper around one existing page (`KioskPage`). It's not a separate product with its own backend, auth system, or independent release cycle.
- It shares the same API base URL, kiosk token auth scheme, and NFC scan endpoint. Keeping it in the same repo means changes to the API contract (e.g. new fields on `/api/nfc-scans`) are visible in one diff.
- The web kiosk page (`client/src/pages/KioskPage.jsx`) continues to exist for non-Samsung devices and QR-only kiosks. The Capacitor app is an additional deployment target, not a replacement.
- Build artifacts (`kiosk-app/android/`) are gitignored. The repo doesn't bloat.
- The earlier mobile app research note (`mobile-app-options-auth-research-2026-03-10.md`) recommended a separate repo for a full React Native rewrite, but explicitly called out Capacitor as a natural extension of the existing project: _"PWA or Capacitor are much more natural extensions of the existing project than React Native... This can be treated as another delivery target rather than a second frontend product."_

If the Capacitor shell later grows into a full worker/employer mobile app with Privy auth, wallet flows, and multiple screens, it may make sense to extract it. But for a single-page kiosk wrapper, same-repo is the right call.

## What Gets Built

### Directory structure

```
kiosk-app/
├── package.json              # React + Vite + Capacitor + NFC plugin
├── capacitor.config.ts       # webDir: "dist", appId, appName
├── vite.config.js            # Standalone Vite build (not the main client/)
├── index.html                # Minimal HTML shell
├── src/
│   ├── main.jsx              # Entry point — renders <App />
│   ├── App.jsx               # Just <KioskPage /> (no router, no Privy)
│   ├── KioskPage.jsx         # Adapted from client/src/pages/KioskPage.jsx
│   └── capacitor-nfc.js      # Thin wrapper around NFC plugin API
├── android/                  # Generated by `npx cap add android`
│   └── ...                   # (gitignored except for key config files)
└── dist/                     # Vite build output (gitignored)
```

### KioskPage adaptation

What stays the same:
- Token setup/reset flow (`localStorage` for kiosk token)
- QR scanning via ZXing `BrowserQRCodeReader`
- `submitNfcBadge()` and `submitToken()` API calls
- Confirmation/error overlay UI
- All Tailwind styling

What changes:
- NDEFReader replaced with Capacitor NFC plugin listener
- No service worker or manifest (native app handles lifecycle)
- No PWA detection logic
- No BroadcastChannel (native app doesn't open new tabs)
- API base URL configured for production (hardcoded or via Capacitor config)

### NFC plugin

The leading candidate is `capacitor-nfc` (community plugin using Android's native NFC adapter). Key API surface:

```javascript
import { Nfc } from 'capacitor-nfc';

// Start listening — uses enableForegroundDispatch() under the hood
await Nfc.startScanSession();

// Listen for tag reads
Nfc.addListener('nfcTag', (tag) => {
  const uid = tag.id; // hex string
  submitNfcBadge(uid, kioskToken);
});
```

If the community plugin has issues, a minimal custom Capacitor plugin (~50 lines of Kotlin) can call `enableForegroundDispatch()` directly.

### Build & distribution

```bash
cd kiosk-app
npm install
npm run build                    # Vite builds to dist/
npx cap sync android             # Copies dist/ into Android project
npx cap open android             # Opens Android Studio
# Build → Generate Signed APK → sideload onto tablet
```

For development iteration:
```bash
npm run build && npx cap sync android && npx cap run android
```

## Risks & Mitigations

| Risk | Mitigation |
|------|------------|
| Community NFC plugin doesn't work on Samsung | Fall back to custom Capacitor plugin (~50 lines of Kotlin calling `enableForegroundDispatch()`) |
| Capacitor WebView has same Samsung NFC interception | Extremely unlikely — native foreground dispatch operates at Activity level, not WebView level. But if so, the custom plugin approach still works because it's pure native code. |
| localStorage not persisted in Capacitor WebView | Use Capacitor Preferences plugin instead of raw `localStorage` (drop-in replacement, backed by SharedPreferences on Android) |
| APK distribution friction | For pilot: sideload via USB or file transfer. Later: Firebase App Distribution or internal Play Store track. |

## What This Enables Later

The Capacitor shell is the foundation for:

1. **GPS oracle** — native geofencing plugin, already planned
2. **Background NFC** — native apps can register for NFC even when backgrounded
3. **Push notifications** — native push for shift reminders
4. **Device management** — Android Enterprise / managed device provisioning
5. **Offline resilience** — queue scans locally when network is down, sync when restored

## Decision

Proceed with building `kiosk-app/` in the existing repo. Web kiosk continues to work for QR-only and non-Samsung devices. The Capacitor app is the NFC solution for Samsung (and likely all future Android kiosk deployments).
